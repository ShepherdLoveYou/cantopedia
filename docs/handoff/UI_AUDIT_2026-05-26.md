## Findings

### P0 — blockers (broken feature, crash, can't navigate)

- _None._

### P1 — visible regressions (layout glitch, theme inconsistency, broken image)

- **[desktop+mobile][all 3 locales] 404 page — "Return home" CTA tile is hard-coded `var(--m-red)`, not accent.** Evidence: `desktop_en_notfound.png`, `desktop_zh_notfound.png`, `desktop_yue_notfound.png`, `mobile_en_notfound.png`, `mobile_zh_notfound.png`, `mobile_yue_notfound.png`. This is exactly the "red where accent should be" pattern that the three previously-shipped color-consistency fixes targeted. Source: `site/src/components/MetroEmptyState.astro` line 27 — `style="background: var(--m-red);"` should be `var(--accent)` (or the CTA tile should use the standard tile-medium accent style). Single fix, 6 affected screenshots.

- **[desktop+mobile][en + yue] 404 page — CTA label "返回首頁" not localized.** Evidence: same 6 notfound screenshots. The 404 page (`site/src/pages/404.astro`) hard-codes `actionLabel="返回首頁"` and `actionHref={base}/zh` regardless of the current locale. On EN and yue 404s the button still reads `返回首頁` and sends users to the `/zh/` home, not `/en/` or `/yue/`. (Note: Astro static site only ships one `404.astro`, so true per-locale routing would need a different mechanism — but at minimum the label should be bi/tri-lingual like the subtitle already is, e.g. `Home · 首頁 · 主頁`.)

- **[desktop+mobile][all 3 locales] Search input on `?q=` URLs renders white text on white background (dark mode).** Evidence: `desktop_en_search-results.png`, `desktop_zh_search-results.png`, `desktop_yue_search-results.png`, `mobile_en_search-results.png`, `mobile_zh_search-results.png`, `mobile_yue_search-results.png` — all six show a totally blank-white input box even though the probe loaded `/search?q=chicken`. The query string IS read into `q.value` and the input is auto-focused, but the focus style in `site/src/pages/[locale]/search.astro` is `background: var(--t-ink); color: #fff` — in dark mode `--t-ink` resolves to `#FFFFFF`, so text becomes white-on-white and invisible. Likely intended `background: var(--t-card)` (or any non-`--t-ink` surface), or `color: var(--t-bg)` to invert correctly.

- **[mobile][all 3 locales] dish-detail page has horizontal overflow.** Evidence: `mobile_zh_dish-detail.png` (overflow 414px), `mobile_yue_dish-detail.png` (414px), `mobile_en_dish-detail.png` (608px — content is ~1.5× the 390px viewport wide). The English locale is dramatically worse than zh/yue, which means the regression is being amplified by longer English ingredient names ("Salt-and-pepper seasoning mix", "Green and red bell pepper") interacting with one of: the inline ingredient `<table>` inside a panel whose `overflow-x:hidden` is set on `.pivot-tab-panel` but NOT on `.pivot-page` itself, the meta-cards `grid-template-columns: repeat(2, 1fr)` (1fr can grow past viewport when grid items contain unbreakable English compounds), or the `.pivot-page-nav` flex row holding two uppercase letter-spaced dish names that exceed `space-between` budget. Recommend reproducing with devtools (Elements → check which descendant has `scrollWidth > clientWidth`) before patching, then add `min-width: 0` or explicit `overflow-x: hidden` at the right ancestor.

### P2 — nits (cosmetic, suggestion, low-impact)

- **[desktop+mobile][all 3 locales] Empty-state stat tiles show bare `0`.** Evidence: `desktop_en_home.png`, `desktop_zh_home.png`, `desktop_yue_home.png`. Two stat tiles on home — "draft" and "ingredients only" — render `0` centered with no visual cue that zero is the *correct/healthy* state. Reads as "broken counter" at a glance. Consider showing a check-glyph, dash, or hiding the tile when count is zero.

- **[mobile][zh] Top-nav "EN" locale chip overlaps the bottom of the brand strip.** Evidence: `mobile_zh_home.png`, `mobile_zh_notfound.png` etc. — the `EN` highlight chip in the locale switcher sits noticeably below the `中 粵 EN` row baseline, creating a slight visual hop. Cosmetic — the chip placement looks fine on desktop.

- **[desktop+mobile][all] Footer copyright width inconsistent.** Evidence: `desktop_en_home.png` (footer panel is narrow, ~70% of viewport, centered) vs `desktop_en_search-empty.png` (same narrow). The full-bleed comment in `BaseLayout.astro` line 912–913 says "Full-bleed via negative margin so it breaks out of main's max-width" — but on the actual screenshots the footer is clearly NOT full-bleed; it stops at the 1200px main width. Either intentionally changed since that comment, or the negative-margin is being clipped by an ancestor. Low priority — looks fine, just internally inconsistent with the comment.

- **[desktop][all 3 locales] Dish hero banner image positioning crops the food photo at top-left.** Evidence: `desktop_zh_dish-detail.png`, `desktop_en_dish-detail.png`, `desktop_yue_dish-detail.png`. `background-position: center` works for square/wide images but the shrimp photo is centered on a thick rim of shrimp — the visible crop on the wide banner ends up showing just shrimp legs, not the dish presentation. Consider per-dish `background-position` overrides, or switch to an `<img>` with `object-position` for art-direction control.

## Out-of-band observations

- **Pagefind missing in dev mode is expected.** All 6 search-results pages show `console error: Failed to load resource: 404` + `netFail: /cantopedia/pagefind/pagefind.js`. Pagefind only writes its bundle during `pnpm build`. The search.astro `runSearch()` catch block already shows a friendly fallback message ("Search index not available in dev. Run `pnpm build`."). No action needed.

- **Intentional 404 visits are expected.** The probe deliberately requests `/<locale>/this-page-does-not-exist` to capture the 404 page render — so each of those 6 page-loads correctly reports `consoleErrors: 3, netFails: 3` (the navigation itself, plus Astro's dev-mode prefetch attempts hitting the dev server's 404 handler). The visual issue with that page is captured in P1 above.

- **`category` page never captured.** The probe tries to follow the first `a[href*="/categories/"]` on home, but in the current home layout, category tiles use `/<locale>/all` or dish detail links rather than `/<locale>/categories/<id>`. If category-page coverage is wanted in the audit, extend the probe to construct a category URL deterministically from `getCollection('category')`.

- **Mobile dish hero "FUMIKAS SAGISAVAS" credit text overlaps the hero name** on the EN locale (`mobile_en_dish-detail.png` — the photo credit caption sits very close to the hero text bottom). Tight but not actually overlapping at this dish; could become a real issue for dishes with longer credits. Low-grade P2 in advance.

- **HubPivot mobile tap-target padding (44×44 invisible from task 4) is in place** — verified visually on the mobile screenshots; chevrons sit far enough from the title that the invisible hit area should not collide with the title. No new finding.

- **Color-consistency fixes from earlier commits hold up.** Search input border = accent ✓ (`desktop_en_search-empty.png`), footer left stripe = accent blue ✓, pivot active underline = accent ✓ (`mobile_en_dish-detail.png` shows "Ingredients" underlined in accent), `a:hover` → accent ✓ (not directly observable in static screenshots but no stray red spotted on hover-style elements). The 404 CTA above is the one place the audit found that still ships red where accent would belong.
